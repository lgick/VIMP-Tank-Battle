# Расширение игры

Гайды по добавлению контента. Общее правило проекта: новые сущности выполняются в едином стиле с существующими (фиксированных контрактов нет — существующие файлы служат образцом), а любое изменение завершается зелёными `npx eslint .` и `npm test` с покрытием нового кода тестами.

## Новая карта

1. Создайте `src/data/maps/<имя>.js` по образцу существующих (например [pool_mini.js](../../src/data/maps/pool_mini.js)). Формат:
   - `setId` — snapshot-ключ конструктора карты (`c1`/`c2`);
   - `scale` — масштаб карты;
   - `spriteSheet` — изображение тайлов и кадры `[x, y, w, h]`;
   - `layers` — распределение тайлов по слоям отрисовки (1 — под танками, 2 — уровень танков, 3+ — над);
   - `physicsStatic` — номера тайлов, являющихся стенами (по ним строится статическая физика и клиентский raycast);
   - `physicsDynamic` — динамические физические объекты (двигаются, передаются в снапшоте);
   - `step` — размер тайла;
   - `respawns` — точки возрождения по командам: массивы `[x, y, угол]`;
   - `map` — матрица тайлов.
2. Зарегистрируйте карту в [src/data/maps/index.js](../../src/data/maps/index.js) — ключ объекта станет названием в голосованиях и для `VIMP_MAP`.

## Новое оружие

Существует два архитектурно разных типа (см. [server.md](server.md#физические-части-srcserverparts)):

- **Hitscan** (пример `w1`): попадание рассчитывается мгновенно лучом через `HitscanService`; физического снаряда нет — только результат.
- **Explosive** (пример `w2`): в мире Rapier создаётся физический снаряд (`Bomb`), живёт в физическом цикле, передаётся клиенту как сущность снапшота, взрывается по таймеру.

Шаги:

1. Определите оружие в [src/data/weapons.js](../../src/data/weapons.js) (тип, урон, кулдаун, расход и т.д.).
2. Реализуйте серверную часть в `src/server/parts/` по аналогии с существующим оружием того же типа.
3. Создайте клиентский рендеринг в `src/client/parts/`.
4. Зарегистрируйте сущность в `src/config/client.js`: `parts.gameSets` (snapshot-ключ → классы) и `parts.entitiesOnCanvas` (класс → полотно).
5. Зарегистрируйте snapshot-ключи оружия (и его эффектов) в `SNAPSHOT_KEYS` в [src/config/opcodes.js](../../src/config/opcodes.js) — незарегистрированный ключ уронит упаковку кадра (`SnapshotPacker.packBody` бросает ошибку). Если существующие `kind` не подходят под формат данных — добавьте новую раскладку блока в [src/lib/snapshotCodec.js](../../src/lib/snapshotCodec.js) и поднимите версию формата.
6. Последним элементом данных события/сущности передавайте **id автора** (как `shooterId` у `w1` и `ownerId` у `w2`) — по нему `ShotPredictor` подавляет серверные дубли клиентского спавна; типы `hitscan`/`explosive` он поддерживает автоматически по конфигу оружия.
7. Добавьте боезапас в `game.js` (`panel`) и ключ панели в `client.js` (`modules.panel`).

## Новый звук

1. Добавьте запись в [src/config/sounds.js](../../src/config/sounds.js): `file`, `priority`, `volume`, опционально `loop`.
2. Положите аудиофайл в `public/sounds/` в форматах **`.webm` и `.mp3`** (список кодеков — `codecList`).
3. Воспроизведение: UI/системные — `soundManager.playSystemSound(name)`; пространственные — `registerSound(name, { position })` (лимит голосов и приоритеты соблюдает `SoundManager`, см. [client.md](client.md#soundmanager)).

## Новая клиентская сущность (part)

1. Создайте класс в `src/client/parts/` по образцу существующих (`Tank`, `Bomb`, эффекты в `parts/effects/`) и экспортируйте в `parts/index.js` — он попадёт в реестр `Factory`.
2. Пропишите её в `gameSets`/`entitiesOnCanvas` (`src/config/client.js`).
3. Если нужна процедурная текстура — добавьте baker в `src/client/providers/bakers/` (ориентируйтесь на существующие) и запись в `bakedAssets`.
4. Если нужны сервисы (`renderer`, `soundManager`) — добавьте класс в `componentDependencies`.

Сущности можно наследовать и отображать на разных полотнах: например, для радара создаётся упрощённый класс (как `MapRadar` от `Map`).

## Тесты

Новый код покрывается тестами в `tests/` (структура зеркалит `src/`). Паттерны — в CLAUDE.md (раздел Testing): синглтоны через `vi.resetModules()` + динамический import, физические части — с моками `world`/`body`, интеграционные сценарии — в `tests/server/integration/`. При изменении модели движения танка обязателен прогон паритет-теста `TankPredictorParity`.
